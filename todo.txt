# Fix: readiness reason string formatting + REST threshold

## 1. Float formatting in reason string — `source/Readiness.mc`

`sample.data` returns a Float, so `"BB " + bb` prints as "BB 39.000000".
Cast to integer before building any reason string.

FIND the `bodyBattery()` function return line:
```monkeyc
return sample.data;
```
REPLACE WITH:
```monkeyc
return sample.data.toNumber();
```

That's it — `bb` is now an Integer everywhere it's used in `assess()` and
the reason string prints "BB 39 low" as intended.

---

## 2. REST threshold is too aggressive — `source/Readiness.mc` + `source/Config.mc`

Current logic: BB 20–39 = SHORT, then a single knockdown (stress / activity /
manual / same-day session) pushes SHORT → REST. At BB 39 that's very easy to
trigger. Two options — pick one:

### Option A (recommended): require BB < 25 before REST is reachable from knockdowns
Change the SHORT band threshold so only very low BB puts you close to REST:
```monkeyc
// FIND:
} else if (bb >= 20) {
    band     = Config.READINESS_SHORT;
    bbReason = "BB " + bb + " low";
} else {
    band     = Config.READINESS_REST;
    bbReason = "BB " + bb + " very low";
}
// REPLACE WITH:
} else if (bb >= 25) {
    band     = Config.READINESS_SHORT;
    bbReason = "BB " + bb + " low";
} else {
    band     = Config.READINESS_REST;
    bbReason = "BB " + bb + " very low";
}
```
With this change, BB 39 = SHORT and would need TWO knockdowns to reach REST
(SHORT → TRIM → REST), which is much harder to hit accidentally.

### Option B: require two knockdowns to reach REST
Change `bandDown` so it only nudges by 1, and only send REST when band
is already SHORT AND there are 2+ knockdown conditions. This is more
complex — Option A is simpler and achieves the same thing.

---

## Acceptance criteria
- Reason string shows "BB 39 low" not "BB 39.000000 low".
- A BB of 39 with a single knockdown (e.g. same-day session) shows
  SHORT suggestion rather than a rest day.
- REST day is reserved for BB < 25 or multiple simultaneous knockdowns.
